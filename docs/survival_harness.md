# Paper Survival Harness

## Purpose

Model the PDF's survival pressure without live trading:

```text
if virtual_bankroll <= death_threshold:
  agent_state = DEAD
  stop opening new positions
  stop forecast spending
```

This is paper accounting only. It does not submit orders, sign messages, move assets, or require a wallet.

## Forecast Provider Interface

Implemented providers:

- `recorded`: uses `fair_yes` already recorded by the forward logger. Current data is neutral market-implied.
- `rule_baseline`: zero-cost baseline using YES order-book midpoint. It is for schema and replay validation, not alpha.
- `synthetic_edge`: test-only provider that injects a fixed edge so the survival engine can exercise position opening.
- `forecast file`: survival can replay CSV/JSONL forecasts generated by `forecast_runner`.

Future providers can reuse the same output fields:

- model name
- fair YES probability
- forecast cost
- reasoning or trace summary

Run with a conservative entry-depth guard:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.survival \
  --input-dir data/paper/live_snapshots \
  --out-dir data/paper/survival_synthetic_depth \
  --bankroll 50 \
  --forecast-mode synthetic_edge \
  --synthetic-edge 0.12 \
  --liquidity-model top3_ask \
  --max-depth-fraction 0.25
```

`top3_ask` uses the paper logger's recorded top-3 ask depth for the selected side. If the proposed entry budget exceeds `top3 ask notional * max_depth_fraction`, the run emits `SKIP_LIQUIDITY_DEPTH` instead of opening the position. This is still a coarse liquidity guard, not full order-book level-by-level simulation.

Allow partial entries inside the same top-3 ask depth guard:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.survival \
  --input-dir data/paper/live_snapshots \
  --out-dir data/paper/survival_synthetic_partial_entry_exit_depth \
  --bankroll 50 \
  --forecast-mode synthetic_edge \
  --synthetic-edge 0.12 \
  --liquidity-model top3_ask \
  --max-depth-fraction 0.25 \
  --entry-fill-policy partial
```

Default `--entry-fill-policy all_or_none` preserves the earlier behavior. With `partial`, the engine opens only the notional that fits inside `top3 ask notional * max_depth_fraction` and records `partial_entries` plus `unfilled_entry_notional`.

Run with a conservative exit-depth guard:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.survival \
  --input-dir data/paper/live_snapshots \
  --out-dir data/paper/survival_synthetic_exit_bid_depth \
  --bankroll 50 \
  --forecast-mode synthetic_edge \
  --synthetic-edge 0.12 \
  --exit-policy edge_below \
  --exit-edge-threshold 0.20 \
  --exit-liquidity-model top3_bid \
  --max-exit-depth-fraction 0.25
```

`top3_bid` uses the paper logger's recorded top-3 bid depth for the selected side. If no bid depth is recorded, the exit is skipped with `SKIP_EXIT_LIQUIDITY_DEPTH`. If only part of the position fits inside the configured bid-depth fraction, the event is `PARTIAL_EXIT_POSITION` and the remaining shares stay open.

Add an optional depth-utilization slippage penalty:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.survival \
  --input-dir data/paper/live_snapshots \
  --out-dir data/paper/survival_synthetic_slippage \
  --bankroll 50 \
  --forecast-mode synthetic_edge \
  --synthetic-edge 0.12 \
  --liquidity-model top3_ask \
  --max-depth-fraction 0.25 \
  --entry-fill-policy partial \
  --exit-policy edge_below \
  --exit-edge-threshold 0.20 \
  --exit-liquidity-model top3_bid \
  --max-exit-depth-fraction 0.25 \
  --slippage-model depth_utilization \
  --max-slippage-bps 50
```

Default `--slippage-model none` preserves earlier results. `depth_utilization` applies up to `max_slippage_bps` linearly by requested notional divided by the configured top-3 depth notional: buys receive a worse ask and sells receive a worse bid. This is intentionally coarse; it does not replay every order-book level or model queue position.

`depth_utilization` requires at least one depth model: `--liquidity-model top3_ask` for entry slippage or `--exit-liquidity-model top3_bid` for exit slippage. The survival runner raises an error instead of silently running with zero slippage when both depth models are disabled.

Avoid false mark-to-zero drawdowns when a snapshot has a temporarily missing bid:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.survival \
  --input-dir data/paper/live_snapshots \
  --out-dir data/paper/survival_synthetic_last_valid_bid \
  --bankroll 50 \
  --forecast-mode synthetic_edge \
  --synthetic-edge 0.12 \
  --missing-quote-policy last_valid_bid
```

Default `--missing-quote-policy zero` preserves earlier conservative behavior. `last_valid_bid` uses the latest valid bid observed for the same market and side only when the current bid field is empty or unparsable. It applies to mark-to-market accounting, early-exit edge checks, and exit execution only when `--exit-liquidity-model top3_bid` is enabled. It does not invent a price before any valid quote has been observed.

Skip paid forecasts when a row cannot trigger a new entry or exit:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.survival \
  --input-dir data/paper/live_snapshots \
  --out-dir data/paper/survival_synthetic_actionable_forecasts \
  --bankroll 50 \
  --forecast-mode synthetic_edge \
  --synthetic-edge 0.12 \
  --forecast-cost 0.01 \
  --forecast-call-policy actionable
```

Default `--forecast-call-policy always` preserves earlier accounting and calls the forecast provider for every non-resolved row. `actionable` skips the provider when the market is already held and `--exit-policy none`, or when a new market cannot be opened because `max_positions` is already full. The result includes `forecast_calls` so cost-sensitive runs can distinguish rows processed from paid model calls.

Reduce same-timestamp path dependence from `market_id` ordering:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.survival \
  --input-dir data/paper/live_snapshots \
  --out-dir data/paper/survival_synthetic_position_first \
  --bankroll 50 \
  --forecast-mode synthetic_edge \
  --synthetic-edge 0.12 \
  --timestamp-order-policy position_first
```

Default `--timestamp-order-policy sequential` preserves earlier row order. `history_first` keeps chronological order but, within the same timestamp, processes markets that have appeared in prior timestamps before brand-new markets. `position_first` is stronger: at the start of each timestamp batch, all batch quotes are first written into the mark-to-market state, then currently held markets are processed before unheld markets, so exits can free slots before new entries are evaluated. This is still an ordering policy, not a full simultaneous batch engine.

Compress MDD accounting to the last equity mark per timestamp:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.survival \
  --input-dir data/paper/live_snapshots \
  --out-dir data/paper/survival_synthetic_timestamp_close \
  --bankroll 50 \
  --forecast-mode synthetic_edge \
  --synthetic-edge 0.12 \
  --timestamp-order-policy position_first \
  --drawdown-policy timestamp_close
```

Default `--drawdown-policy event` preserves earlier max-drawdown accounting and includes every intra-row equity mark. `timestamp_close` keeps event logs unchanged but calculates the headline `max_drawdown` from the last marked equity within each timestamp, reducing false MDD caused by arbitrary same-timestamp event ordering. Reports always include both `event_max_drawdown` and `timestamp_close_max_drawdown`; use the event value for conservative survival-risk review because timestamp-close accounting can hide intra-timestamp stress.

Summarize all survival reports into one CSV:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.report_summary \
  --input-dir data/paper \
  --output-csv data/paper/survival_report_summary.csv
```

The summary loader excludes `latest_survival_report.json` by default to avoid duplicate rows. Legacy reports that do not contain `event_max_drawdown` or `timestamp_close_max_drawdown` are still loaded; both split MDD fields fall back to the old `max_drawdown`, and `legacy_mdd_fields=true` marks the weaker evidence. The CSV includes `run_timestamp`, `initial_bankroll`, `return_on_investment`, entry/exit liquidity fields, and `missing_fields` so older reports are not mistaken for fully modeled zero-slippage or zero-liquidity-risk runs.

Compare the latest run per scenario:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.strategy_compare \
  --summary-csv data/paper/survival_report_summary.csv \
  --output-csv data/paper/survival_strategy_comparison.csv \
  --output-md docs/survival_strategy_comparison.md
```

The comparison keeps only the latest `run_timestamp` per `scenario` by default and marks `mixed_history=true` when a scenario folder contains materially different run signatures. `screening_rank` is a triage heuristic: ALIVE before DEAD, scenarios with opened positions before no-trade baselines, higher ROI, then lower event MDD. It should be used to choose review targets, not to assert a deployable trading strategy.

For current-schema-only comparison, rerun candidate scenarios under `data/paper/standardized/...` and filter the table:

```bash
PYTHONPATH=src .venv/bin/python -m polymarket_backtest.strategy_compare \
  --summary-csv data/paper/survival_report_summary.csv \
  --scenario-prefix standardized/ \
  --rank-mode triage \
  --output-csv data/paper/survival_standardized_comparison.csv \
  --output-md docs/survival_standardized_comparison.md
```

Use `--rank-mode performance` when the table should sort by financial outcome rather than active-strategy debugging priority. In performance mode, a no-trade baseline with 0% ROI can rank above active strategies that lost money; in triage mode, active strategy runs rank above no-trade baselines so execution failures can be debugged first.

## Current Interpretation

With `recorded` mode, no positions should open because current `fair_yes` is market-implied and all logged rows have edge below the PDF's 8 percentage point threshold.

The useful next step is replacing `rule_baseline` with real timestamped forecast providers, then comparing cost-adjusted survival and eventual resolved-market outcomes.

## Review Notes

Antigravity sanitized review was run after implementation. Codex accepted these points:

- Current results are mechanics checks, not evidence that the agent can survive live markets.
- Synthetic edge still loses mark-to-market value because entries pay ask and open positions are marked to bid. This is useful because it exposes spread drag.
- `max_positions=10` intentionally blocks further entries after the first 10 positions; future versions need exit/resolution handling before more realistic capital recycling.
- Forecast cost should eventually be paid only when the forecast provider is actually called after pre-screening, not blindly for every row.
- Resolution cashflow and early-exit policy are now implemented. The next material engine work is attaching real forecast providers and collecting enough post-entry resolution outcomes to evaluate realized PnL.
- A zero-spread exit regression test now verifies that the synthetic early-exit loss disappears when bid equals ask, supporting the spread-drag interpretation.
- Edge comparison uses a small epsilon to avoid accidental floating-point boundary behavior.

Additional forecast-harness review fixes:

- Due resolutions are now checked against all open positions whenever any later row is processed, so a resolved market can settle even if that market no longer appears in active paper logs.
- Empty quote/liquidity fields are converted safely to `0.0` instead of crashing the run.
- Naive `resolved_at` timestamps are treated as UTC for comparison.
- Unknown resolution outcomes are modeled as cost refunds and labeled `UNKNOWN`; this is a conservative paper-accounting placeholder until official cancellation/refund semantics are wired in.
- Forecast costs no longer drive cash negative. If a paid forecast cost would exhaust available cash, cash is set to zero and the agent dies.
- Optional `top3_ask` liquidity guard now prevents entry budgets from exceeding a configurable fraction of recorded top-3 ask depth.
- Broad Antigravity review found a max-drawdown distortion when the agent died on the first row. The survival equity curve now starts at the initial bankroll, so instant death reports `max_drawdown=1.0` instead of `0.0`.
- Death labels are now separated as `cash_depleted_by_forecast_cost`, `equity_below_death_threshold`, and `equity_depleted_after_position`.
- High-model Antigravity review found timestamp ordering and forecast replay key risks. Paper rows are now sorted by parsed timestamp rather than raw string order, and forecast-file keys normalize equivalent UTC timestamps such as `Z` and `+00:00`.
- Forecast cost accounting now records only the actually paid cash on insolvency. If a forecast costs more than available cash, `forecast_cost_total` increases by remaining cash, not the unaffordable quoted cost.
- Exit liquidity now has an optional `top3_bid` guard. `unfilled_exit_notional` is current unresolved exit exposure by market, not a repeated per-snapshot sum. The guard is still a coarse model because it uses aggregate top-3 depth at the best-bid price rather than level-by-level fills.
- Entry liquidity now supports `--entry-fill-policy partial`, making the entry side symmetric with partial exits when desired. It is disabled by default to preserve conservative all-or-none entry behavior.
- Complete entry skips under `entry-fill-policy partial` now add the skipped budget to `unfilled_entry_notional`.
- Stale `unfilled_exit_notional` is cleared when an exit signal disappears or when the affected position later settles.
- Partial entries do not yet scale in to a target Kelly size on later snapshots; existing positions still block additional entry for the same market. Treat this as a conservative sizing limitation.
- Optional `--slippage-model depth_utilization` now records `slippage_cost_total` and worse executed prices when configured entry or exit depth is heavily used. This still remains a top-3 aggregate approximation, not a full order-book simulator.
- Optional `--missing-quote-policy last_valid_bid` prevents one missing bid snapshot from immediately marking an open position to zero. Exit execution with this fallback requires `--exit-liquidity-model top3_bid` to avoid phantom exits without a depth check. Use the default `zero` mode when intentionally testing worst-case quote disappearance.
- Death-threshold checks now run immediately after exits and settlement, not only after the next forecast loop. This avoids delayed death states after a large realized loss.
- Optional `--forecast-call-policy actionable` avoids paid forecast calls for rows that cannot act because max positions are full or because an existing position has no exit policy enabled. This is a cost-control approximation; it still calls the provider before edge evaluation for potentially tradable rows.
- Actionable forecast skips still append current marked equity and apply the death threshold before continuing, so cost-control skips cannot hide a drawdown or delayed death state.
- Optional `--timestamp-order-policy history_first` reduces same-timestamp `market_id` path dependence by processing previously seen markets before new markets. A later full batch engine should split each timestamp into update, exit, then entry phases.
- Optional `--timestamp-order-policy position_first` further reduces same-timestamp path dependence by pre-updating all quotes in a timestamp batch, then processing currently held markets before unheld markets. A later full batch engine should split each timestamp into explicit update, settlement, exit, then entry phases.
- Optional `--drawdown-policy timestamp_close` keeps event logs unchanged but calculates headline MDD from only the last equity mark inside each timestamp. This is useful when multiple markets share one logger timestamp and row-level event order should not be interpreted as a real sequence of tradeable drawdowns. Reports still include `event_max_drawdown` so intra-timestamp stress is visible.
- `polymarket_backtest.report_summary` reads both old and new survival JSON reports. Treat `legacy_mdd_fields=true` rows as less precise because their event-vs-timestamp-close MDD split is reconstructed from the old single `max_drawdown`.
- `polymarket_backtest.strategy_compare` ranks latest scenario runs for review. Treat `mixed_history=true` rows cautiously because old experiments in the same scenario folder may not represent one stable strategy configuration.
